# Style Guide

This document defines the writing style of **The IE Field Guide**.

## Language

The main language of this project is English.

A Chinese version may be considered in the future, but the current project should focus on producing a clear, readable, and practical English edition first.

## Tone

The tone should be:

- Practical
- Honest
- Student-friendly
- Reflective
- Clear but not overly formal
- Serious without sounding official
- Personal without becoming self-centered

The guide should not sound like a university document. It should feel like a thoughtful senior student explaining the path to future students.

## Writing Principles

1. Explain why before explaining what.
2. Give routes, not just isolated tips.
3. Use examples whenever possible.
4. Mark personal experience as personal experience.
5. Avoid pretending that one path works for everyone.
6. Avoid unnecessary anxiety.
7. Prefer clarity over academic decoration.
8. Be honest about uncertainty.
9. Do not turn personal opinions into universal rules.
10. Help readers build judgment, not dependence.

## Preferred Voice

Preferred phrasing:

> In my experience, ...

> One possible way to think about this is ...

> This may not apply to everyone, but ...

> A useful mental model is ...

> If you are just starting out, it may help to ...

Avoid overly absolute phrasing:

> You must ...

> Everyone should ...

> The only correct way is ...

> If you do not do this, you will fail ...

## Terminology

Use standard English technical terms.

Examples:

- Information Engineering
- Computer Science
- Data Structures
- Algorithms
- Systems
- Signal Processing
- Networking
- Research
- Internship
- Version Control
- Software Engineering
- Mathematical Foundation

Technical terms should be explained when they first appear if the target reader may not know them.

## Chapter Structure

Each chapter should ideally include:

1. A short opening paragraph
2. The main problem or question
3. Key ideas
4. Practical advice
5. Common mistakes
6. Suggested next steps

Not every chapter needs to follow this structure rigidly, but the reader should always know:

- What this chapter is about
- Why it matters
- How to use the advice
- What to do next

## Personal Experience

Personal experience is allowed, but it should not be presented as universal truth.

The guide may include reflections, mistakes, lessons learned, and subjective observations. However, these should be framed carefully.

Good:

> In my experience, starting projects early helped me understand programming beyond coursework.

Better:

> In my experience, starting small projects early helped me understand programming beyond coursework. This may not be the best path for everyone, but it is one useful route.

Avoid:

> Everyone should start projects immediately.

## AI-Assisted Writing Policy

This project may use AI tools during the writing process.

AI may be used for:

- Brainstorming
- Outlining
- Language polishing
- Rewriting rough notes
- Improving clarity
- Checking consistency
- Generating alternative phrasings

AI should not be used to:

- Invent personal experience
- Fabricate facts
- Create fake citations
- Replace final human judgment
- Present unreviewed text as finished content

All AI-assisted content must be reviewed before publication.

Preferred disclosure phrase:

> AI-assisted, human-directed.
